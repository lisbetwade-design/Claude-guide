01 Foundations

What is Claude?

Claude is an AI assistant made by a company called Anthropic. You type to it. It writes back. That's the whole interface — a conversation.

But what makes Claude different from a chatbot or a search engine is that it actually understands what you're asking and responds with real thought. It reads context, holds a conversation, builds on what you've said, and can tackle almost any creative or analytical task.

For designers, it's the collaborator you never had: one who can critique your work honestly, draft copy instantly, explain technical things in plain language, write working code, and never get tired of your revisions.

💬

Conversational

Type like you're messaging a colleague. No commands, no syntax. Just describe what you need.

🧠

Contextual

Remembers everything in your conversation. Builds on previous answers, picks up on nuance.

🔁

Iterative

Revise endlessly. Push back, change direction, ask for three more versions. It won't get frustrated.

👁️

Visual

Upload screenshots, mockups, wireframes. Claude can look at them and give you real, specific feedback.

⚡

Fast

What takes you an hour to research and draft, Claude does in seconds. The speed is genuinely shocking at first.

🏗️

Generative

Doesn't just answer — it creates. Copy, code, strategy, documentation, briefs. Full outputs, not just ideas.

02 The Mindset Shift

How to think about Claude

Most beginners get mediocre results from Claude not because they ask the wrong things — but because they have the wrong mental model. They treat it like a vending machine: put in a request, get out an answer.

The designers who get incredible results treat Claude like a thinking partner. The conversation itself is the design process. You give context, Claude responds, you refine, it builds, you redirect. Back and forth until you have something real.

❌ Wrong mental model

"Give me a design for a login screen."

Treating Claude like a one-shot generator. You get generic output, feel disappointed, and assume AI is overhyped.

✓ Right mental model

"Let's figure out this login screen together."

You give context. Claude responds. You push back, redirect, refine. The output gets better with every exchange.

❌ Wrong mental model

Treating the first response as final.

Taking whatever Claude gives first and moving on. You'll miss 80% of its value by not iterating.

✓ Right mental model

The first response is a draft.

The first output shows you what's possible. The real work is the 3–5 messages that follow, shaping it into something great.

❌ Wrong mental model

Asking vague questions, getting vague answers.

"What should I do with this design?" gives you nothing useful. Claude needs context to give useful output.

✓ Right mental model

Brief Claude like you'd brief a designer.

Who is the user? What's the problem? What have you tried? What tone? What constraints? More context = better output.

The most important thing to internalize: Claude is not a replacement for your judgment — it's an amplifier of it. Your taste, your experience, your design instinct is still driving. Claude just removes all the friction around execution.

05 First Session

Your first 30 minutes with Claude

Don't read about Claude. Do this. Open claude.ai in another tab right now and follow these five exercises in order. Each one reveals a different dimension of what's possible.

1
2 minutes

Ask it to explain something you've always been fuzzy on

Pick a technical concept that's come up in your work but you've never fully understood — CSS variables, REST APIs, component tokens, whatever. Ask Claude to explain it like you're a designer who doesn't code.

Try this Explain what CSS variables are, why developers use them, and how I should think about them when I'm designing a token system in Figma. Assume I understand design systems but have never written code.
2
5 minutes

Upload a screenshot of something you're working on

Take a screenshot of any UI — your own work, something you're redesigning, a competitor's app. Upload it to Claude. Ask for honest critique.

Try this [attach screenshot]
Be honest. What are the three most significant problems with this design? Give me specific, actionable feedback — not generic best practices.

Then push: ask it to go deeper on one of the issues it raised. Ask what it would do differently. Ask it to compare this to how a company you admire might solve the same problem.

3
5 minutes

Ask it to write copy — then rewrite it three different ways

Give Claude a real UX writing task from a project you're working on. An empty state, an onboarding headline, a button label, an error message — anything.

Try this I'm working on a [type of app] for [type of user]. Write 5 variations of the headline for the first onboarding screen. The tone should feel [adjective], not corporate. Each under 10 words.

Once you have 5 options, pick the one that's closest to what you want and say: "I like option 3 but it feels too [x]. Rewrite it with [y]." This iteration loop is where Claude really shines.

4
10 minutes

Ask it to build you something in code

This one will surprise you. Ask Claude to build a simple interactive UI component — and watch it generate working HTML, CSS, and JavaScript in under a minute. You don't need to understand the code.

Try this Build a card component for a mobile app. It should show: a user avatar (placeholder), a name, a short status message, and an "Follow" button that toggles to "Following" when clicked. Clean, modern style. Dark background. Just HTML and CSS.

Claude will give you code in its Artifacts panel — a live preview you can interact with right there. Click the follow button. See it work. Then ask it to change the colors, add an animation, make it wider. Watch how fast it responds to your direction.

5
5 minutes

Use it to challenge your own thinking

This is the most underused capability. Claude is a rigorous thinking partner — use it to stress-test your ideas before you're in a client meeting or design review.

Try this I'm designing [describe your current project briefly]. I've decided to [describe your design decision]. What are the strongest arguments against this decision? Don't hold back.

After these five exercises, you'll have a real feel for what Claude can do. The rest of this guide gives you the vocabulary, prompts, and patterns to go further.

08 Common Mistakes

Why beginners get bad results

Most people who try Claude and feel underwhelmed are making one of these five mistakes. Recognizing them is the shortcut to getting great output.

🌫️

Prompts that are too vague

The most common mistake. "Design me a landing page" or "help me with my app" gives Claude nothing to work with. Claude can only be as specific as the context you give it.

"Build a landing page for a B2B invoicing tool targeting freelance designers. Sections: hero with one-line value prop, 3 key features, social proof, and CTA. Tone: professional but not corporate. Dark background."
🛑

Stopping at the first response

The first response is a starting point. Treating it as the final answer is like accepting the first sketch from a designer without any feedback. The real quality comes through iteration.

After any response: "Good start. Now make option 2 more [quality]. Also try a version that takes a completely different angle." Always push for at least one round of refinement.
🔄

Starting a new chat instead of continuing

Every new chat starts fresh — Claude forgets all previous context. Beginners start new chats for every question. Instead, keep related work in one long conversation. Claude builds on everything you've shared.

Keep a single long conversation for each project or task. At the start, set context once: "I'm working on X for Y users. Let me share my thinking first before we start generating..."
📋

Copying output without editing

Claude's output is raw material — not finished product. Especially for copy and documentation, the first pass needs your editorial voice applied. Claude removes 80% of the work; you do the last 20% that makes it distinctly yours.

Treat everything Claude writes as a first draft. Read it, edit it, make it sound like you. The goal is to use Claude to get to a great draft 10× faster — not to bypass your judgment entirely.
🔮

Asking Claude what to build instead of how to build it

"What should I design for my portfolio?" or "What features should my app have?" — Claude will answer, but these are the wrong questions. You're the designer. Strategy and direction come from you. Claude executes and enhances.

Bring Claude a decision you've already made, not a blank slate. "I've decided to build X for Y users. Help me [write copy / review this design / build this component]." Your judgment drives; Claude accelerates.
09 Honest Expectations

What Claude can't do

Using Claude well means knowing where it falls short. These aren't reasons to avoid it — they're reasons to keep your judgment engaged.

It doesn't know your users

Claude's design advice is based on general patterns. It hasn't done research on your specific users. Always pressure-test its suggestions against your actual research and context.

It will sometimes be confidently wrong

Claude can "hallucinate" — stating incorrect information with confidence. For facts, statistics, or technical specifics, verify from a primary source before using them.

Its code can have bugs

Generated code usually works but sometimes has issues — especially in complex interactions. Test everything. Ask Claude to fix errors by pasting them back in. It's good at self-correcting.

It doesn't remember past conversations

Each new chat is a blank slate. Projects (Pro feature) give you some persistence, but in general, you need to re-establish context each session.

Its taste isn't always your taste

Claude's design sensibility leans toward safe, modern, and conventional. Push it explicitly toward the aesthetic you want. It responds well to specific references: "more like Linear, less like Salesforce."

It can't browse the internet (without tools)

By default, Claude doesn't know about current events, new design releases, or anything after its training cutoff. Use it for thinking and creating, not for researching what launched last week.

The golden rule: Never publish copy, ship code, or present findings from Claude without reading and editing them yourself first. Claude removes the grunt work — it doesn't replace your professional judgment.

13 The Most Important Habit

Always plan before you build

This is the single most important thing experienced vibe designers have learned — and it's the thing beginners skip. Never ask Claude to start building immediately. Ask it to make a plan first.

Every source agrees on this. Design thinkers who've shipped real products describe it as the difference between a clean build and debugging spaghetti at 2am. When you give Claude a vague instruction and say "go", it will — but it might go in the wrong direction for a long time before you notice.

❌ How beginners prompt

"Build me a portfolio site with a hero, projects grid, and contact form."

Claude immediately starts writing files. You have no idea what it chose or why. When it goes wrong, you don't know where to course-correct.

✓ How experienced builders prompt

"Research the best approach and write a plan.md. Don't write any code until I approve the plan."

Claude thinks first. You review. You adjust. Then it executes — with shared understanding of the direction.

The plan prompt (copy this exactly)

Always use this first I want to build: [describe your project in plain English]

Before writing any code, research the best approach and create a plan.md file with:
1. Recommended tech stack and why
2. File structure
3. Any external services needed
4. Step-by-step implementation plan

Don't write any code until I've reviewed and approved the plan.

14 Project Memory

CLAUDE.md — teach Claude your project

By default, every Claude Code session starts fresh. It doesn't remember your project structure, your design decisions, or your preferences from last time. CLAUDE.md fixes this.

CLAUDE.md is a plain text file you put in your project folder. Claude Code reads it automatically at the start of every session — so it always arrives knowing your project, your rules, and your design principles. Think of it as onboarding a new team member, written down once and reused forever.

For designers: This is where your design philosophy lives. Your color system, your spacing rules, your tone of voice, your brand constraints. Write it once, and every session Claude Code already knows how your project thinks.

What to put in your CLAUDE.md

CLAUDE.md — Designer template
# Project: [Your project name]

## What this is
A [type of app] for [type of user]. The goal is [outcome].

## Design principles
- Visual style: [minimal / bold / editorial / playful]
- Tone of voice: [describe it]
- Reference brands: [2-3 companies whose aesthetic we're aiming for]
- What we avoid: [gradients / Comic Sans / excessive animation / etc.]

The old design workflow was linear: design in Figma → hand off to dev → wait for implementation → review → feedback loop for weeks. The new workflow is a closed loop where design and code inform each other in real time.

As of February 2026, Figma and Anthropic have built direct integrations that make this loop faster than ever. Here's how the pieces fit together.

How to use it

16 Going Live

Deploy your work — free, in minutes

Building something is one thing. Having a live URL you can send to a client, put in your portfolio, or share on social media is another. The deployment stack used by virtually every designer who vibe codes is GitHub + Vercel — both free, both connect together in minutes.

🗂️

GitHub

Where your code lives and gets version-controlled. Like a cloud save + history for your project. Every time you "commit," it saves a snapshot you can revert to.

⚡

Vercel

Turns your GitHub project into a live website. Every push to GitHub auto-deploys. You get a free URL instantly. Custom domain in under 10 minutes.

The full deployment flow

1

Create a GitHub account

Go to github.com, sign up for free. This is where your project will live.

github.com → Sign up free
2

Ask Claude Code to set up git and push your project

Create a new repository on GitHub (click + → New repository), then tell Claude Code to connect and push:

Deployment prompt Initialize git in this project, connect it to my GitHub repo at https://github.com/[yourusername]/[repo-name].git, and push everything. Also create a README.md and a CLAUDE.md describing this project.
3

Deploy to Vercel

Go to vercel.com, sign up with your GitHub account. Click "Add New Project," import your repository, click Deploy. You'll have a live URL in under 2 minutes.

vercel.com → Add New → Import from GitHub → Deploy
4

Every future update auto-deploys

From now on, whenever Claude Code commits to GitHub, Vercel automatically picks it up and deploys the new version. Your live URL always reflects the latest build.

You: The hero button color is wrong, make it #e94560
✓ Updated. Committing and pushing to GitHub...
✓ Vercel deployment triggered automatically
✓ Live at: https://your-project.vercel.app
5

Add a custom domain (optional)

In Vercel: Settings → Domains → Add your domain. Vercel shows you two DNS records to add at your domain registrar. Takes 5–30 minutes to propagate. Your site loads on yourname.com.

Vercel → Settings → Domains → Add domain

One key safety rule: Never commit a file called .env to GitHub. This file contains your API keys. Always ask Claude Code to create a .gitignore file that excludes it — and it will handle this automatically if you ask.
